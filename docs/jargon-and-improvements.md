# Momo — Tech Jargon & Design Rationale

Two audiences, one doc. Each concept is explained **simply** (plain-language), then
**technically** (what actually happens), then the **correlation** — how the simple idea
maps to the technical mechanism and why it's fast.

---

## 1. "The popup opens instantly" → Pre-warmed panel

**Simple:** The little window that shows your clipboard history is built once when the
app starts and then just hidden and re-shown. Pressing the hotkey doesn't *build*
anything — it *reveals* something that's already there. Like a drawer you slide open,
not a table you assemble each time.

**Technical:** The UI lives in an `NSPanel` created at launch with
`isReleasedWhenClosed = false`. The hotkey handler calls `makeKeyAndOrderFront` /
`orderOut` — no view allocation, no Auto Layout pass, no data reload on the open path.
The panel is `.nonactivatingPanel` at `.floating` level so it appears over the active
app without a full app activation.

**Correlation:** "Instant" = zero work on the hot path. Every allocation and layout
computation is paid once at startup, never on open.

> **Typical menu-based managers** rebuild/repopulate their UI when invoked, so open cost
> scales with history size and does layout work on the main thread each time. That
> per-open work is exactly the "popup open delay" you feel. Momo moves that cost to
> launch, once.

---

## 2. "Typing filters instantly" → In-memory search index + off-hot-path work

**Simple:** When you type in the search box, Momo searches a copy of your history
that's already sitting in memory — it never goes to disk while you type. Searching
1000 items this way takes microseconds.

**Technical:** `HistoryIndex` holds the recent ~1000 items as lightweight structs
(id, preview, kind, timestamp). `search(query:)` runs a subsequence fuzzy match over
that array and re-sorts by `(pinned, matchScore, recency)`. The database (`Store`) is
never queried on a keystroke. The table diff-updates rather than fully rebuilding.

**Correlation:** "Instant typing" = the search path touches only RAM and only cheap
computation. Disk I/O and heavy work are structurally impossible on that path because
the index has no reference to the database.

> **Typical menu-based managers** can jank on large histories. Momo's hard separation —
> `HistoryIndex` (speed, RAM-only) vs `Store` (durability, disk, background-only) —
> makes a slow keystroke architecturally impossible.

---

## 3. "It notices what you copy without draining your battery" → changeCount polling

**Simple:** macOS doesn't tell apps "hey, the clipboard changed." So Momo peeks at a
tiny counter a few times a second. Only when that counter moves does it actually read
the clipboard. Peeking is nearly free.

**Technical:** `NSPasteboard` exposes `changeCount` (an integer that increments on every
write) but emits no notifications. A `DispatchSourceTimer` on a background utility queue
fires every ~250ms and compares `changeCount` to the last seen value. Reading the actual
contents happens *only* on a change. All of this runs off the main thread.

**Correlation:** "Doesn't drain battery" = the frequent operation (reading one integer)
is trivial, and the expensive operation (reading + persisting content) is rare and
backgrounded. Idle CPU stays near zero.

**Jargon:**
- **`changeCount`** — monotonically increasing pasteboard revision number.
- **Polling** — checking state on an interval because there's no event to subscribe to.
- **QoS (Quality of Service)** — the priority hint (`.utility`) that tells macOS this
  work is background and can yield to the UI.

## 4. "It won't save your passwords" → Privacy filter on pasteboard hints

**Simple:** Password managers mark what they copy as "secret / don't remember me."
Momo honors that mark and never stores those items.

**Technical:** `PrivacyFilter.shouldStore(pasteboardTypes:)` rejects any pasteboard
carrying `org.nspasteboard.ConcealedType`, `TransientType`, or `AutoGeneratedType`
(the community-standard markers 1Password and others set). The check runs before an
item is ever built or written.

**Correlation:** "Won't save secrets" = a pure, first-line gate on the ingest path.
Because it's a pure function over the type set, it's trivially unit-tested and can't be
accidentally bypassed downstream.

**Jargon:**
- **UTI / pasteboard type** — a reverse-DNS string identifying what kind of data is on
  the clipboard (e.g. `public.utf8-plain-text`).
- **Concealed / transient type** — convention flags meaning "sensitive" / "ephemeral."

## 5. "Copying the same thing twice doesn't clutter the list" → Dedupe-to-top

**Simple:** Copy the same text again and it just jumps back to the top instead of
adding a duplicate row.

**Technical:** Every item has a deterministic `contentHash` (djb2 over a canonical
payload string). `Store.upsert` looks up that hash; if found, it updates the existing
row's `createdAt` (bumping recency) instead of inserting. `HistoryIndex.prepend`
mirrors this in memory by removing any same-hash entry before inserting at the front.

**Correlation:** "No clutter" = identity is content-based, not time-based. The same
rule is enforced in both the durable store and the in-memory index so they never
disagree.

**Jargon:**
- **Upsert** — insert-or-update in one operation.
- **Content hash** — a fixed-length fingerprint of the payload used as an identity key.

## 6. "Big images don't slow anything down" → Blob-on-disk + lazy thumbnails

**Simple:** Screenshots are stored as files on disk, not inside the searchable list.
The list only loads a small thumbnail, and only when that row is actually on screen.

**Technical:** Image bytes are written via `Store.writeImageBlob` to
`Application Support/Momo/images/<uuid>.png`; the DB row keeps only the relative path
and byte size. `HistoryRowView.configure` loads the thumbnail on a background queue and
sets it on the main queue, and only for visible (reused) rows in the virtualized
`NSTableView`.

**Correlation:** "Images don't slow things down" = image weight never enters the search
path (index holds a path string, not pixels) and never blocks the UI (thumbnails load
async, off-screen rows load nothing).

**Jargon:**
- **Blob** — Binary Large OBject; raw bytes stored outside the structured columns.
- **Cell reuse / virtualization** — the table only creates views for visible rows and
  recycles them as you scroll, so a 1000-row list costs ~10 rows of work.
- **Lazy loading** — fetch a resource only at the moment it's needed.

---

## 7. "It pastes into whatever app you were in" → Focus stash + synthetic Cmd+V

**Simple:** When you open Momo, it remembers which app you were using. After you pick
an item, it puts that app back in front and simulates pressing Cmd+V for you.

**Technical:** On show, `PanelController` stashes
`NSWorkspace.shared.frontmostApplication`. On choose, `Paster.writeToPasteboard` sets
the chosen item, the panel hides, the stashed app is re-activated, and after an ~80ms
focus settle, `Paster.synthesizePaste` posts a synthetic ⌘V via a `CGEvent` on the
HID event tap.

**Correlation:** "Pastes where you were" = the non-activating panel never truly steals
ownership, so restoring focus is just re-activating the remembered app; the synthetic
keystroke completes the paste the way you'd do it by hand.

**Jargon:**
- **Non-activating panel** — a window that can receive input without making its app the
  active/foreground app.
- **`CGEvent` / HID event tap** — low-level API to synthesize and post input events
  (keystrokes) system-wide. Requires the user to grant Accessibility permission.

---

## 8. Why native AppKit, not SwiftUI / Electron

**Simple:** We use the oldest, closest-to-the-metal Mac UI toolkit because it gives us
direct control over exactly when work happens. Web-based apps (Electron) ship a whole
browser and are heavy — the opposite of the goal.

**Technical:** AppKit lets us own the panel lifecycle, table virtualization, and
run-loop timing explicitly. SwiftUI's declarative diffing can itself become the source
of open/typing latency on list-heavy UIs, which is precisely the failure mode we're
eliminating. Electron/Tauri add a Chromium/WebView runtime with startup and memory
overhead unacceptable for an always-resident menubar utility.

**Correlation:** The whole product thesis is "control the hot path." AppKit maximizes
that control; the alternatives abstract it away.

---

## Summary table

| You feel | Momo mechanism |
|---|---|
| Popup opens instantly | Pre-warmed `NSPanel` (build once) |
| Typing filters instantly | RAM-only `HistoryIndex` + fuzzy match |
| No battery drain | Background `changeCount` poll |
| Passwords never stored | `PrivacyFilter` ingest gate |
| No duplicate clutter | Content-hash dedupe-to-top |
| Images don't lag | Disk blobs + lazy thumbnails + virtualization |
| Pastes where you were | Focus stash + synthetic ⌘V |

> Momo's edge is narrow and deliberate: move every expensive operation off the two
> paths a human actually waits on — the popup opening, and a search keystroke.
